# CURE project stages

## One project, two design stages

The research target is **CURE**: a general correction framework placed after a
frozen IRSTD detector through a thin adapter. **CURE-Lite is Stage 1 of CURE.**
It is a deliberately minimal mechanism test, not a separate final project and
not a reduced substitute for the intended CURE method.

The current hypothesis is:

> Uniformly deleting one legal covered component from the Base occupancy,
> while keeping the frozen feature fixed, can teach a positive residual
> predictor to add factual-miss targets under shared false-alarm and retention
> constraints.

Stage 2 is Full CURE design. It begins only after Stage 1 has real support. The
additional Full CURE mechanism must be derived from a measured Lite mismatch;
it is not chosen in advance and is not implemented in the active package.

## Active Stage 1 boundary

The active method consists of frozen-Base contracts, occupancy, matching,
intervention, factual and synthetic supervision, the residual decoder, its
loss, monotone union, calibration, and fixed-budget evaluation. Experiment
orchestration lives under `cure_lite.experiment` and does not change the method
definition.

The Base is required as an experimental input, but no detector architecture is
part of CURE-Lite. The formal Stage-A runner starts from two generic cache
indexes. Every record supplies:

- the frozen Base probability map `p_b`;
- one detached feature map `F_b`;
- sample, preprocessing, grid, and Base identities needed for consistent use.

For the first real pilot, those caches are generated by a project-owned
reference U-Net trained only on `D_B`. The reference U-Net is not a CURE module,
not a claimed innovation, and not imported by the Stage-A core or CLI.

The occupancy-to-feature projection uses adaptive max pooling. A source-grid
legal deletion is eligible for synthetic training only if the projected
occupancy seen by the decoder changes. Stage-A v2 also performs two checks
before any decoder update:

1. the selected Base anchor must satisfy the configured total false-alarm and
   covered-target-retention constraints;
2. the real `D_R` cache must meet the declared `support_requirements`, including
   minimum factual states, reachable misses, synthetic images, and
   projected-visible legal candidates.

The observed support is recorded in `receipts/support.json`.

## Stage 1 evidence sequence

1. Validate the method semantics and software path.
2. Validate the deterministic toy mechanism and its known failure case.
3. Use a group-disjoint `D_B / D_R / D_V` development manifest; use only
   `D_B-fit / D_B-select` for reference-Base training and checkpoint choice.
4. Train the project-owned reference U-Net for the configured 800 epochs.
5. Export model-independent `D_R / D_V` `(p_b, F_b)` caches.
6. Run the single-seed `A / Base@B / F / U` real Stage-A pilot.
7. Treat all `D_V` numbers only as development selection scores and decide
   whether the minimal mechanism has enough support to continue.

The latest complete software run reports `149 passed`. This closes the tested
software path but does not replace steps 4–7 and does not prove a real-data
gain.

The four-way pilot answers three distinct questions:

- `U > A`: does the residual route improve on the selected Base anchor?
- `U > Base@B`: is the result more than relaxing the Base threshold?
- `U > F`: does uniform legal intervention add evidence beyond factual-only
  residual learning?

`F` and `U` share factual examples and their order, but `U` has additional
synthetic exposure. A paper-level attribution to the intervention therefore
also needs an exposure-matched factual control (`F×`) with the same number of
updates/examples as `U`. The current four-way Stage-A remains a first screen.

No Stage-1 command reads `D_T` images or labels. `D_T` evaluation belongs only
to a later, explicitly prepared evaluation phase after the development choices
are fixed.

## Transition to Full CURE

CURE-Lite success is the entry condition for Full CURE design:

- If the projected-visible legal intervention does not outperform `Base@B`,
  threshold relaxation is a sufficient explanation; revise the minimal
  mechanism before proceeding.
- If it does not outperform factual learning and the exposure-matched control,
  the intervention has no independent support; revise it before proceeding.
- If it improves held-out factual misses within all configured constraints,
  characterize the remaining synthetic-to-factual or cross-detector mismatch.
  That measured mismatch becomes the design requirement for Full CURE.

Full CURE will retain the common Base-evidence interface and behave as a
post-detector plugin. Thin adapters for DNANet, UIUNet, MSHNet, and SCTransNet
will translate native preprocessing/output/features to the shared contract;
the CURE core must not be rewritten per detector. Multi-backbone validation
starts only after the Full CURE mechanism is defined.

Historical MSHNet-specific files document an earlier integration attempt. They
do not define the active Base, formal Stage-A path, or current project stage.
